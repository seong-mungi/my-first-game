# Foundation/Core ADR 단계 쉽게 설명

이 문서는 코딩 전에 작성해야 하는 6개의 핵심 ADR을 비전문가 기준으로 풀어쓴 설명입니다.

**ADR(Architecture Decision Record)**은 “왜 이런 구조로 만들기로 했는가?”를 남기는 기술 결정 문서입니다.  
여기서 다루는 6개 ADR은 게임 기능을 직접 만드는 일이 아니라, **누가 무엇을 책임지는지 미리 정하는 교통정리 작업**입니다.

---

## 한눈에 보기

| ADR | 쉽게 말하면 | 핵심 질문 |
|---|---|---|
| 1. Scene lifecycle and checkpoint restart architecture | 씬 전환과 재시작 담당 정하기 | 스테이지 시작/전환/재시작은 누가 관리하나? |
| 2. Cross-system signal architecture and event ordering | 시스템 간 알림 규칙 정하기 | 총 발사, 피격, 사망 같은 사건을 누가 알리고 누가 듣나? |
| 3. Save and settings persistence boundary | 저장 범위 정하기 | 무엇을 파일에 저장하고, 무엇은 현재 실행 중에만 유지하나? |
| 4. Input polling, pause handling, and UI focus boundary | 입력과 메뉴 조작 규칙 정하기 | 같은 버튼이 게임 조작인지 메뉴 조작인지 누가 판단하나? |
| 5. Player entity composition and lifecycle state ownership | 플레이어 구성과 상태 소유자 정하기 | 플레이어 위치, 무기, 탄약, 죽음 상태는 누가 책임지나? |
| 6. Damage, HitBox/HurtBox, and combat event ownership | 전투 판정 책임 정하기 | 무엇이 맞았고 죽었는지는 누가 최종 판정하나? |

## 1. Scene lifecycle and checkpoint restart architecture

### 의미

**스테이지를 시작하고, 바꾸고, 죽었을 때 다시 시작하는 흐름을 정하는 ADR**입니다.

### 정하는 것

- 게임 시작 시 어떤 씬을 불러올지
- 플레이어가 죽으면 어느 체크포인트에서 다시 시작할지
- 체크포인트 위치는 누가 기억할지
- 씬 전환 전/후에 어떤 신호를 보낼지
- 재시작을 1초 안에 끝내는 규칙을 어떻게 지킬지

### 쉬운 비유

무대 감독입니다.  
배우, 조명, 음악이 각자 마음대로 무대를 바꾸면 혼란스럽기 때문에, **무대 전환은 SceneManager가 맡는다**고 정하는 단계입니다.

---

## 2. Cross-system signal architecture and event ordering

### 의미

**게임 시스템들이 서로 어떻게 알림을 주고받을지 정하는 ADR**입니다.

### 예시

플레이어가 총을 쏘면 여러 시스템이 반응합니다.

- 총알 생성
- 사운드 재생
- 카메라 흔들림
- HUD 탄약 표시 변경
- VFX 표시

이때 총 시스템이 다른 시스템을 직접 조종하지 않고, **“총이 발사됨”이라는 신호를 보내고 필요한 시스템들이 듣는 구조**를 정합니다.

### 추가로 정해야 하는 것

같은 순간에 여러 일이 생길 때의 순서도 정해야 합니다.

- 플레이어가 죽는 순간 보스도 죽었다면?
- 총알이 맞는 순간 씬 전환이 시작됐다면?
- 되감기와 데미지가 같은 프레임에 발생했다면?

### 쉬운 비유

학교 방송 시스템입니다.  
선생님이 모든 교실을 직접 뛰어다니는 대신 방송으로 알리고, 각 교실이 자기 역할에 맞게 반응하는 방식입니다.

---

## 3. Save and settings persistence boundary

### 의미

**무엇을 저장하고, 무엇은 저장하지 않을지 정하는 ADR**입니다.

### 정하는 것

- 세이브 파일에 저장할 것
- 설정 파일에 저장할 것
- 게임을 껐다 켜도 유지할 것
- 현재 실행 중에만 유지할 것
- 저장 시스템이 없을 때 다른 시스템이 몰래 파일을 쓰지 못하게 하는 규칙

### 예시

- 볼륨 설정은 나중에 저장 가능
- Tier 1의 임시 볼륨 조정은 현재 실행 중에만 적용
- Time Rewind snapshot은 저장 파일로 쓰면 안 됨
- 체크포인트 상태를 저장할지는 별도 정책 필요

### 쉬운 비유

집안 금고 규칙입니다.  
아무나 중요한 서류를 금고에 넣거나 빼면 안 되므로, **무엇을 저장할 수 있고 누가 저장 책임자인지** 정합니다.

---

## 4. Input polling, pause handling, and UI focus boundary

### 의미

**키보드/게임패드 입력, 일시정지, 메뉴 선택을 누가 처리할지 정하는 ADR**입니다.

### 정하는 것

- 이동/점프/공격 입력은 언제 읽을지
- pause 버튼은 어떤 상태에서 동작하는지
- 죽는 중이나 되감기 중에는 pause를 막을지
- 메뉴에서 현재 선택된 버튼 focus를 누가 관리할지
- 게임패드와 키보드 입력이 동시에 들어오면 어떻게 처리할지
- UI 조작과 gameplay 입력이 섞이지 않게 하는 방법

### 예시

- 게임플레이 입력은 physics tick에서만 읽음
- pause/menu 입력은 UI 쪽 예외로 처리 가능
- 메뉴 focus는 Menu/Pause UI가 관리
- HUD는 보여주기만 하고 focus를 받지 않음

### 쉬운 비유

리모컨 규칙입니다.  
게임 중에는 버튼이 캐릭터를 움직이고, 메뉴 중에는 같은 버튼이 메뉴 선택을 움직입니다. **언제 어떤 의미인지** 미리 정해야 합니다.

---

## 5. Player entity composition and lifecycle state ownership

### 의미

**플레이어 캐릭터가 어떤 부품으로 구성되고, 각 상태를 누가 관리할지 정하는 ADR**입니다.

### 플레이어를 이루는 부품 예시

- 이동 담당
- 무기 담당
- 피격 판정
- 애니메이션
- 상태 머신
- 되감기용 snapshot 데이터

### 정하는 것

- 플레이어 루트 노드는 무엇인지
- 무기는 플레이어 안에 붙는지
- 탄약 수는 누가 소유하는지
- 살아있음/죽는 중/되감기 중 상태는 누가 관리하는지
- 되감기 복원 시 어떤 값들을 되돌리는지
- 어떤 시스템이 플레이어 위치를 직접 바꾸면 안 되는지

### 쉬운 비유

자동차 설계도입니다.  
엔진, 핸들, 브레이크, 계기판이 각각 있지만 아무 부품이나 속도 값을 마음대로 바꾸면 위험합니다. 그래서 **각 부품의 책임과 권한**을 정합니다.

---

## 6. Damage, HitBox/HurtBox, and combat event ownership

### 의미

**무엇이 맞았고, 누가 데미지를 판정하며, 전투 이벤트를 누가 소유하는지 정하는 ADR**입니다.

### 용어

- **HitBox**: 공격 판정 영역  
  예: 총알이 닿으면 데미지를 주는 영역
- **HurtBox**: 맞는 판정 영역  
  예: 플레이어나 적이 공격을 받을 수 있는 영역

### 정하는 것

- 총알이 적에게 맞았는지 누가 판정하는지
- 플레이어가 적 공격에 맞았는지 누가 판정하는지
- 한 방에 죽는 규칙은 누가 소유하는지
- 같은 프레임에 여러 총알이 맞으면 어떻게 처리하는지
- 보스 사망 신호는 누가 발행하는지
- 데미지 판정과 VFX/Audio가 섞이지 않게 하는 방법

### 중요한 원칙

VFX는 폭발을 보여줄 수 있지만 데미지를 결정하면 안 됩니다.  
Audio는 피격음을 낼 수 있지만 죽음을 결정하면 안 됩니다.

### 쉬운 비유

심판입니다.  
관중이 환호하고 조명이 번쩍여도, 실제로 점수가 났는지는 **심판(Damage system)**이 결정합니다.

---

## 최종 요약

이 6개 ADR은 게임 기능 자체가 아니라, 기능을 안전하게 만들기 위한 **책임 분담 문서**입니다.

먼저 책임자를 정하면:

- 코드가 덜 꼬이고
- 테스트하기 쉬워지고
- 나중에 시스템을 추가할 때 충돌이 줄고
- “이건 누가 해야 하지?” 같은 애매함이 줄어듭니다.

**한 줄 요약:**  
게임을 만들기 전에 교통법규를 정하는 단계입니다. 차량이 달리기 전에 신호등, 차선, 우선순위를 먼저 정하는 것과 같습니다.

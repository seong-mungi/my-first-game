# Foundation/Core ADR 단계 쉽게 설명

아래 6개는 **코딩 전에 정해야 하는 “기술 설계 결정서(ADR)” 목록**입니다.

쉽게 말하면:

> 게임을 만들 때 중요한 책임 분담과 규칙을 미리 정해서, 나중에 코드가 뒤엉키지 않게 하는 단계입니다.

---

## 1. Scene lifecycle and checkpoint restart architecture

### 쉽게 말하면

**스테이지를 시작하고, 바꾸고, 죽었을 때 다시 시작하는 일을 누가 책임질지 정하는 단계**입니다.

### 정하는 내용

- 게임 시작 시 어떤 씬을 불러올지
- 플레이어가 죽으면 어디서 다시 시작할지
- 체크포인트 위치는 누가 기억할지
- 씬 전환 전/후에 어떤 신호를 보낼지
- 재시작이 1초 안에 끝나야 한다는 규칙을 어떻게 지킬지

### 비유

무대 감독입니다.

> 배우, 조명, 음악이 마음대로 무대를 바꾸면 난장판이 되니, “무대 전환은 무대 감독(SceneManager)만 한다”라고 정하는 것입니다.

---

## 2. Cross-system signal architecture and event ordering

### 쉽게 말하면

**게임 안의 여러 시스템들이 서로 어떻게 알림을 주고받을지 정하는 단계**입니다.

### 정하는 내용

예를 들어 플레이어가 총을 쏘면 여러 일이 생깁니다.

- 총알 생성
- 사운드 재생
- 카메라 흔들림
- HUD 탄약 표시 변경
- VFX 표시

이때 서로 직접 건드리지 않고:

> “총이 발사됨”이라는 신호를 보내고, 필요한 시스템들이 그 신호를 듣는다

는 방식으로 설계합니다.

또한 같은 순간에 여러 이벤트가 발생하면 순서도 정해야 합니다.

예:

- 플레이어가 죽는 순간 보스도 죽었다면?
- 총알이 맞는 순간 씬 전환이 시작됐다면?
- 되감기와 데미지가 같은 프레임에 발생했다면?

### 비유

학교 방송 시스템입니다.

> 담임 선생님이 모든 교실에 직접 뛰어다니는 게 아니라, 방송으로 알리고 각 교실이 알아서 반응하는 방식입니다.

---

## 3. Save and settings persistence boundary

### 쉽게 말하면

**무엇을 저장하고, 무엇은 저장하지 않을지 정하는 단계**입니다.

### 정하는 내용

- 세이브 파일에 저장할 것
- 설정 파일에 저장할 것
- 게임을 껐다 켜면 유지할 것
- 현재 실행 중에만 유지할 것
- 저장 시스템이 아직 없을 때 다른 시스템이 몰래 파일을 쓰지 못하게 막는 규칙

예:

- 볼륨 설정: 나중에는 저장 가능
- 현재 일시정지 메뉴의 임시 볼륨 조정: Tier 1에서는 세션 전용
- Time Rewind snapshot: 저장 파일로 쓰면 안 됨
- 체크포인트 상태: 누가 저장할지 정해야 함

### 비유

집안 금고 규칙입니다.

> 아무나 중요한 서류를 금고에 넣거나 빼면 안 되니, “무엇을 저장할 수 있고, 누가 저장 책임자인지” 정하는 것입니다.

---

## 4. Input polling, pause handling, and UI focus boundary

### 쉽게 말하면

**키보드/게임패드 입력을 누가 읽고, 일시정지와 메뉴 조작을 어떻게 처리할지 정하는 단계**입니다.

### 정하는 내용

- 이동/점프/공격 입력은 언제 읽을지
- pause 버튼은 어떤 상태에서 먹히는지
- 죽는 중이나 되감기 중에는 pause를 막을지
- 메뉴에서 현재 선택된 버튼 focus를 누가 관리할지
- 게임패드와 키보드 입력이 동시에 들어오면 어떻게 처리할지
- UI 때문에 gameplay 입력이 꼬이지 않게 하는 방법

예:

- 게임플레이 입력은 physics tick에서만 읽음
- pause/menu 입력은 예외적으로 UI 쪽에서 처리 가능
- 메뉴 focus는 Menu/Pause UI가 관리
- HUD는 보여주기만 하고 focus를 받지 않음

### 비유

리모컨 규칙입니다.

> 게임 중에는 리모컨 버튼이 캐릭터를 움직이고, 메뉴 중에는 같은 버튼이 메뉴 선택을 움직입니다. 언제 어떤 의미인지 미리 정해야 합니다.

---

## 5. Player entity composition and lifecycle state ownership

### 쉽게 말하면

**플레이어 캐릭터가 어떤 부품으로 구성되고, 살아있음/죽는 중/되감기 중 같은 상태를 누가 관리할지 정하는 단계**입니다.

### 정하는 내용

플레이어는 단순한 이미지 하나가 아니라 여러 부품입니다.

- 이동 담당
- 무기 담당
- 피격 판정
- 애니메이션
- 상태 머신
- 되감기용 snapshot 데이터

이 단계에서는 다음을 정합니다.

- 플레이어 루트 노드는 무엇인가?
- 무기는 플레이어 안에 붙는가?
- 탄약 수는 누가 소유하는가?
- 죽는 중 상태는 누가 관리하는가?
- 되감기 복원 시 어떤 값들을 되돌리는가?
- 어떤 시스템이 플레이어 위치를 직접 바꾸면 안 되는가?

### 비유

자동차 설계도입니다.

> 엔진, 핸들, 브레이크, 계기판이 각각 있지만 아무 부품이나 속도 값을 마음대로 바꾸면 위험합니다. 누가 어떤 상태를 책임지는지 정하는 것입니다.

---

## 6. Damage, HitBox/HurtBox, and combat event ownership

### 쉽게 말하면

**무엇이 맞았고, 누가 데미지를 판정하며, 전투 이벤트를 누가 소유하는지 정하는 단계**입니다.

### 용어 쉽게 설명

- **HitBox**: 공격 판정 영역  
  예: 총알이 닿으면 데미지를 주는 영역

- **HurtBox**: 맞는 판정 영역  
  예: 플레이어나 적이 공격을 받을 수 있는 영역

### 정하는 내용

- 총알이 적에게 맞았는지 누가 판정하는가
- 플레이어가 적 공격에 맞았는지 누가 판정하는가
- 한 방에 죽는 규칙은 누가 소유하는가
- 같은 프레임에 여러 총알이 맞으면 어떻게 처리하는가
- 보스 사망 신호는 누가 발행하는가
- 데미지 판정과 VFX/Audio가 섞이지 않게 하는 방법

중요한 원칙:

> VFX는 폭발을 보여줄 수 있지만, 데미지를 결정하면 안 됩니다. Audio는 피격음을 낼 수 있지만, 죽음을 결정하면 안 됩니다.

### 비유

심판입니다.

> 관중이 환호하고 조명이 번쩍여도, 실제로 점수가 났는지는 심판(Damage system)이 결정합니다.

---

## 전체 흐름 요약

이 6개는 모두 **코딩 전에 “책임자와 규칙”을 정하는 작업**입니다.

| 단계 | 핵심 질문 |
|---|---|
| 1. Scene lifecycle | 씬 전환과 재시작은 누가 관리하나? |
| 2. Signal/event | 시스템들은 어떻게 서로 알리나? |
| 3. Save/settings | 무엇을 저장하고 누가 저장하나? |
| 4. Input/UI | 입력과 메뉴 focus는 누가 처리하나? |
| 5. Player entity | 플레이어 상태와 부품은 누가 소유하나? |
| 6. Damage/combat | 맞음/죽음/전투 이벤트는 누가 판정하나? |

## 한 줄 요약

**이 6단계는 게임 기능을 만드는 작업이 아니라, 기능을 안전하게 만들기 위한 “교통정리 문서”를 작성하는 작업입니다.**  
누가 무엇을 책임지는지 먼저 정해야, 나중에 코드가 꼬이지 않고 테스트도 쉬워집니다.
